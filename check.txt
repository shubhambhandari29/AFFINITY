## Current understanding

`SpecialAccountsToExcel.py` generates loss-run Excel reports for Special Accounts.

The process works like this:

1. It identifies active customers who require loss-run reports.
2. It reads claim information from the existing `SAC_Loss_Run` database view.
3. The database view prepares most of the report data, including policy, claim, reserve, payment, recovery, and expense information.
4. The script separates normal claims from “Record Only” incidents.
5. It opens the existing Excel template.
6. It fills the Claims Data and Record Only worksheets.
7. It updates the customer number, customer name, and report date on the cover page.
8. It prepares the workbook so its summaries and pivot tables refresh when opened.
9. It saves one Excel file for each customer.

The database view prepares the data, while the Python script mainly formats that data into the required Excel report.

## Proposed FastAPI implementation

We can add this functionality to the existing application in the following way:

```text
User selects a customer
        ↓
User clicks “Generate Loss Run”
        ↓
UI calls a FastAPI endpoint
        ↓
API reads that customer’s data from SAC_Loss_Run
        ↓
API fills the existing Excel template
        ↓
Excel file is returned to the user
```

The implementation can be divided into three parts:

- **UI:** Add a “Generate Loss Run” button.
- **API:** Add a protected endpoint that receives the customer number.
- **Service:** Read the loss-run data, create the workbook, and return it for download.

For the first version, we should continue using the existing database view. We only need to change it if testing shows that its data or business rules are incorrect.

## Suggested implementation plan

1. Obtain the official `SACLossRunTemplate.xlsx` file.
2. Confirm that the application can read the `SAC_Loss_Run` view.
3. Create a service that reads loss-run data for one selected customer.
4. Move the reusable Excel-generation logic from the Databricks script into the service.
5. Add a FastAPI endpoint that returns the generated Excel file.
6. Connect the UI button to the endpoint.
7. Compare the new report with an existing report for the same customer.
8. Test customers with:

   - Normal claims
   - Record-only incidents
   - Both types of records
   - No available records
   - Large numbers of claims

## Main challenges

### Excel template dependency

The report depends on an existing Excel template with specific worksheets, tables, formulas, pivots, and charts. We need to store and protect the correct template inside the application.

If worksheet or table names are changed, report generation could fail.

### Pivot tables and calculations

Python fills the workbook, but it does not fully calculate Excel pivot tables. The workbook is marked to refresh when the user opens it in Excel.

We should confirm that this behavior is acceptable.

### Database access

The application must have permission to read the `SAC_Loss_Run` view and all data required by it.

The Databricks script has its own connection setup, while the application uses its existing database configuration. We need to verify that both point to the correct database and environment.

### Report generation time

Some customers may have many claims. Their reports could take longer to generate.

For the first version, the user can wait for the download. If reports take too long, we may later need a background process with a “report is ready” notification.

### File size and memory

The existing script loads data for every customer. The API should load data only for the selected customer to reduce memory usage and make generation faster.

### Data formatting

Fields such as claim number, policy number, customer number, and exposure may contain leading zeros. We must keep them as text so Excel does not change their values.

Currency and date fields must also remain consistent with the existing report.

### Customer names and filenames

Customer names may contain characters that are unsafe in filenames. The API should create a safe filename, preferably using both customer number and name.

### Multiple users

Several users might generate reports at the same time. The current script uses the same temporary filename for every report, which could cause one report to overwrite another.

The FastAPI version should create a separate temporary file or an in-memory workbook for each request.

### Security

The endpoint should use the application’s existing authentication. Users should only be able to generate reports for customers they are permitted to access.

### Business-rule validation

We should confirm the rules used by the database view, particularly:

- How many years of closed claims should be included?
- Should all open claims be included regardless of age?
- What qualifies as a “Record Only” incident?
- How should claims above $50,000 be counted?
- What should happen when a customer has no records?

## Recommended first version

The first version should be kept simple:

- Generate a report for one selected customer.
- Continue using the existing database view.
- Continue using the existing Excel template.
- Return the workbook directly as a download.
- Do not change the view unless report comparison identifies a problem.
- Do not include automatic email distribution yet.

This approach keeps the initial change small and allows us to verify that the application-generated report matches the current Databricks-generated report.
