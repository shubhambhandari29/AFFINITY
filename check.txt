Hi Team,
As part of the SAC platform authentication design, we have evaluated multiple approaches for implementing user login and role-based access control for the application. Based on our analysis, we have identified two possible approaches that can be used to support secure user authentication and authorization.
Below is a summary of both approaches.
Approach 1: Azure AD Group-Based Authorization
In this approach, user authentication is handled through F5 APM integrated with the organization’s enterprise authentication system (Azure AD / Microsoft Entra ID).
The high-level flow would work as follows:
A user accesses the SAC platform URL.
The request is routed through F5 APM, which validates the user using enterprise Single Sign-On.
Once authenticated, F5 extracts the user identifier and passes it to the application.
The SAC React application forwards this user identifier to the backend FastAPI service.
The FastAPI backend then queries Microsoft Entra ID using Microsoft Graph API to determine the Azure AD groups that the user belongs to.
Based on the returned AD group membership, the backend determines the user's application role (for example: Underwriter, Director, Admin, etc.).
The backend generates an authentication token and returns the user profile and role to the application.
In this model, authorization is controlled through Azure AD group memberships, and any role changes would typically be managed through the organization's identity management process.
Key characteristics of this approach:
Authentication handled through F5 + Azure AD
Authorization based on Azure AD group membership
Requires Microsoft Graph API integration
Role changes are managed through AD group administration
Approach 2: Application Database-Based Authorization (Alternate Approach)
In addition to the Azure AD group-based model, we have identified an alternate approach that simplifies the authorization process while still leveraging F5 for secure authentication.
In this model, F5 continues to handle user authentication, but user roles are managed directly within the SAC application database.
The flow would work as follows:
The user accesses the SAC platform through the corporate URL.
F5 APM authenticates the user through the enterprise SSO mechanism.
Once authenticated, F5 passes the user identifier to the React application.
The React application sends the user identifier to the FastAPI backend through a login/bootstrap API.
The backend checks whether the user exists in the SAC application database table (tblUsers).
If the user exists and is active, the backend retrieves the user’s stored role and profile information from the database.
The backend then generates an authentication token and returns the user profile and role to the application.
In this model, authorization is controlled within the application database rather than Azure AD groups.
To support this approach, the SAC application will provide administrative functionality within the existing List Management module, allowing authorized administrators to manage users directly within the application.
Administrators will be able to:
Add new users
Remove existing users
Update user roles
Manage user access records stored in the tblUsers table
Key characteristics of this approach:
Authentication handled through F5
Authorization managed through application database
No dependency on Azure AD group membership or Graph API
User roles managed through application administration UI
Summary
Both approaches provide secure authentication through F5, but they differ in how user roles and access control are managed.
Feature	Azure AD Group Approach	Application Database Approach
Authentication	F5 + Azure AD	F5
Authorization Source	Azure AD Groups	SAC Application Database
Role Management	Managed via AD groups	Managed via application admin UI
External Dependencies	Microsoft Graph API required	No external identity lookup required
We have prepared architecture diagrams for both approaches to help illustrate the flows. These are attached for reference.
Please review the options and let us know if you have any preferences or guidance on which model aligns better with your operational and governance requirements.
