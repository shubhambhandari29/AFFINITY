Do not call backend /auth/login as an AJAX login bootstrap anymore.
Instead, do one of these:
redirect browser to backend Easy Auth login endpoint, or
authenticate on frontend first and then call backend with authenticated session/token
If your frontend and backend are on separate App Services, the common pattern is to initiate login by full-page navigation, not fetch, because interactive sign-in is a browser redirect flow. The CORS troubleshooting guidance from Microsoft also points to redirect/auth flows being a common cause of these browser-side failures.
What to change in code
Current bad pattern
Your React code is probably doing something like:

await fetch("https://sacplatformapipreprd.azurewebsites.net/auth/login", {
  method: "POST",
  credentials: "include",
  headers: { "Content-Type": "application/json" }
})





1) Stop using your custom /auth/login API as the cross-origin login entry point
When Easy Auth is enabled, login should usually be initiated by browser navigation to the Easy Auth endpoint, not by fetch('/auth/login') from React. If you keep calling /auth/login from React as an AJAX call, the browser will keep issuing preflights and Easy Auth will keep getting in the way. App Service Authentication is designed to authenticate the request before your code runs.
Use a full-page redirect from the frontend, for example:

window.location.href =
  "https://sacplatformapipreprd.azurewebsites.net/.auth/login/aad?post_login_redirect_uri=https://sacplatformwebpreprd.azurewebsites.net";



After login succeeds, React should call a lightweight endpoint like /auth/me to read the logged-in user context.


What I’d do next, in order
In React, temporarily stop calling backend /auth/login with fetch.
Trigger login by browser redirect to /.auth/login/aad.
In API App Service, verify built-in CORS includes the exact frontend URL.
Keep FastAPI CORS middleware as well.
Add a simple authenticated endpoint like /auth/me that reads Easy Auth headers.
If business insists on keeping /auth/login, configure Easy Auth path exclusion for that route using file-based auth config and excludedPaths.



Option 2 — Fully switch to Azure App Service / Entra login
This means:
remove or stop using email/password login from React
stop posting to /auth/login
frontend redirects browser to Azure login
FastAPI reads authenticated user from X-MS-CLIENT-PRINCIPAL
then you map that user to your DB role/branch
Microsoft documents that App Service forwards user identity in the X-MS-CLIENT-PRINCIPAL header, which you decode in the app.
In this model
Your React code would no longer do:
api.post("/auth/login", { email, password })
Instead it would do browser redirect login, and your backend would expose something like /auth/me to return current authenticated user info.
Best fit when
This is right only if your Azure team wants Entra/App Service auth to be the real login mechanism.
